Storage


All Moonboard and search data is cached on your local device. The first time you visit (or search), that crag's database will be downloaded and saved locally. All subsequent operations related to that crag will use your local copy of its database. Page templates (crag, area and route/problem) are also cached for up to ten days. Caching everything locally reduces bandwidth usage, provides a faster experience on poor connections and allows you to use Beta offline. To use Beta offline, simply visit the crags you want to use offline along with at one area and one route so that all page templates are cached. Download on WiFi at home and you're good to go at the crag!

The crag databases are lightweight (the vast majority are less than 500kb uncompressed) and automatically update in the background, so there is no need for you to worry about managing your storage.

Search


The search bar allows you to search for a crag, area, route or problem by name, providing matching results in real time. You can also use a variety of commands to filter the results by crag, grade, type or number of pitches. Search results are pulled from your device's local storage so you never have to wait.

Commands

  /  
Activate the search bar if it's not currently focused.
  @g  
Set the search to the global scope.
  @$crag  
Set the search scope to the specified $crag.
  $grade  
Show routes/problems matching the YDS/Hueco $grade. 
  <$grade  
Show routes/problems no harder than the YDS/Hueco $grade. 
  >$grade  
Show routes/problems at least as hard as the YDS/Hueco $grade. 
  p$pitches  
Show routes with the exact number of $pitches. 
  <p$pitches  
Show routes with no more than the number of $pitches. 
  >p$pitches  
Show routes with at least the number of $pitches. 
  -$type  
Show routes/problems tagged with the specified $type.
  \$type  
Show routes/problems tagged with the specified $type.
  ?$type  
Show routes/problems tagged with the specified $type.
  !$type  
Show routes/problems that are not tagged with the specified $type.

Examples

  @g  
Show all crags, routes and problems.
  @g happy  
Show crags, routes and problems with 'happy' in their name.
  -s >5.9-  
Show sport routes graded 5.9- and harder at the current crag.
  -t p1 >5.8  
Show single-pitch trad routes graded 5.8 and harder at the current crag. 
  >5.8 <5.10  
Show routes graded between 5.8 and 5.10d. 
  @g -t <5.10  
Show trad routes graded 5.10d and easier at all crags (global scope). 
  @g -s -a  
Show routes that are tagged sport and aid at all crags. 
  @sm -s !t  
Show sport routes not tagged as trad at Smith Rock. 
  @sm -s >p2  
Show multi-pitch sport routes at Smith Rock.

Scope

By default, the search is scoped to the current crag, or global if there is no valid crag, e.g. searching from this help page will default to the global scope. The search's scope can be changed using the @$crag command, where $crag is the scope shortcut of the crag you want to search, e.g. @sm for Smith Rock. The global scope can be activated via @g. Scope shortcuts for each crag can be found on the crag's main page. To keep the results readable, only crags and routes are shown at global scope, areas are omitted.

Grade

Search results can be filtered to a specific grade simply by typing in the grade, e.g. 5.9 or v6. For grades 5.10 and above the letter grade can be omitted to show all routes in the numeric grade range, e.g. 5.12 will show 5.12a, 5.12b, 5.12c and 5.12d routes, whereas 5.12c will only show 5.12c routes. The grade command only recognizes YDS and Hueco grades.

To show a range of grades, prefix the grade with < or > to specify a minimum or maximum grade respectively. You can specify both a minimum and maximum grade (separate commands) to show a specific range of grades, e.g. >5.10 <5.11 to show all routes between 5.10a and 5.11d. Note that the minimum and maximum grades are inclusive, i.e. <5.9 includes 5.9 in the results.

Pitches

Search results can be filtered to show routes with a specific number of pitches using the p$pitches command, where $pitches is the number of pitches to filter by, e.g. p1 to show only single pitch routes. As boulder problems do not have a pitches property, the pitches command will automatically exclude all boulder problems.

Like the grade command, prefix the pitch command with < or > to specify a minimum or maximum number of pitches respectively. You can specify both a minimum and maximum number of pitches (separate commands) to show a specific range, e.g. >2 <5 to show all routes with at least two pitches but no more than five pitches. Note that the minimum and maximum number of pitches are inclusive, i.e. <3 includes routes with three pitches in the results.

Type

Search results can be filtered to include only specific types of routes via the type command -$t, where $t is a single character type shortcut, e.g. -t to include only trad routes in the results. To exclude a specific type, use the anti-type command !$t, e.g. !s to exclude sport routes from the results. The type shortcuts for both including and excluding types are:

  b
Boulder
  s
Sport
  t
Trad
  a
Aid
  r
Top Rope

To provide easier-to-reach key commands on various smartphone keyboards, the type command recognizes several alternative syntaxes to the dash: ? and \, e.g. ?s or \s to show only sport routes. There is no alternative syntax for the anti-type command.

Up to two (anti-)type commands can be specified to restrict the results to routes that satisfy both type commands, e.g. -t -a will match only routes that are tagged as both Trad and Aid routes, and -s !t will show sport routes while excluding routes that are also tagged as trad, i.e. don't show sport routes that require gear for some sections.

Go


The Go API provides the ability to perform inline searches in your browser's location bar. Go does not return a list of results but rather redirects to the first matching result, à la Google's "I'm Feeling Lucky". Go is available at http://moon.zombull.xyz/go?q=%s, where %s is the query to perform a search on. To redirect to the associated Mountain Project page instead of the Beta page, add the query string parameter mp=1 to the request.

Go currently does not support any specialized commands, though areas and route names are prefixed with their crag name so searches can be scoped to some extent by prepending some or all of the crag name. Basic wildcards are supported (? for a single character, * for zero to N characters) and are useful when prefacing the search with the crag name. The search will look for matches anywhere in the crag/area/route name so wildcards are only necessary when specifying non-contiguous portions of the name. Similar to the in-app search, results are prioritized to return crags first, areas second and routes last.

Search Engines

The Go API is intended to be used for custom search engines in your browser (instructions for Chrome and Firefox). For example, you might bind the keyword b to http://moon.zombull.xyz/go?q=%s, and the keyword mp to http://moon.zombull.xyz/go?mp=1&q=%s.
